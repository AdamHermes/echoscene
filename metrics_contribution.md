# Physics-Based Verification and Metrics Contribution (ProcTHOR / AI2-THOR)

## Overview of the Contribution

While generative models for 3D room layouts (such as our core diffusion model) rely heavily on analytical, heuristic Python functions to guide placement (e.g., 2D bounding box intersection penalties, Gaussian heatmaps, and spatial constraints), these heuristic losses alone are not sufficient to guarantee functional realism. Python-based array checks cannot natively simulate the complex, non-linear realities of a 3D physical environment—such as agent capsule radius, continuous pathing constraints, or functional accessibility.

To bridge the gap between simple generative mathematical constraints and realistic functional evaluation, our work introduces a **rigorous simulation-based verification framework**. While the constraints fundamentally reduce to a 2D continuous pathfinding problem, we leverage ProcTHOR (built on the AI2-THOR Unity engine) as a highly robust, standardized tool to perform these evaluations. 

By utilizing an established embodied AI simulator, we gain access to out-of-the-box continuous NavMesh baking, precise agent-radius clearance checks, and industry-standard navigation APIs. We propose two primary evaluation metrics—**Walkability** and **Navigability (Accessibility)**—that validate the functional usability of the generated layouts.

---

## 1. Bridging the Generative-to-Physical Gap

During generation, our model relies on physical guidance (as defined in our Python codebase) to minimize collisions and encourage traversable central spaces. However, verifying these results requires more than just checking bounding box overlaps. 

**The Verification Pipeline:**
1. **Layout Translation:** We export the generative model outputs (`final.json`) which contain 2D/3D bounding boxes, orientations, and scales.
2. **Procedural Instantiation:** Using `convert_echoscene_to_procthor.py`, these generated layouts are mapped precisely into the physics engine. The furniture footprints are instantiated as rigid, solid obstacles, explicitly projecting the scene into a rigorous continuous pathing environment.
3. **Agent Simulation & Pathfinding:** An AI2-THOR agent is securely spawned into the room (using a 2D Euclidean Distance Transform algorithm to find the safest spawn coordinate) to probe the environment.

This pipeline ensures that we are not just evaluating simple 2D intersection math, but rather **solving a continuous 2D pathfinding simulation where the agent has a strict physical radius (capsule constraint) that cannot squeeze through unrealistic gaps.**

---

## 2. The Walkability Metric (Free Space Analysis)

The first metric we contribute is **Walkability**, which measures how much of the generated room is physically traversable.

### How it Works
Instead of simply subtracting the 2D area of furniture from the room's total area, this metric uses the Unity **NavMesh** (`GetReachablePositions`). The physics engine floods the floor space and returns a discrete list of valid grid points where the agent can physically stand.

### Mathematical Rigor
- The environment is evaluated at a grid resolution of `0.25 meters` (each point = $0.0625 m^2$).
- The calculation accounts for the agent's physical radius ($\sim 0.3m$). Because the agent has a physical volume, it cannot stand flush against walls or objects. 
- **Walkability Score = (Walkable Area) / (True Total Area)**

### Why it Matters
This verifies that the diffusion model is generating spacious, easily traversable rooms rather than simply packing objects tightly. A room with perfectly non-overlapping objects might still be unusable if the objects form a maze with gaps smaller than $0.6m$ (choke-points). The Walkability metric accurately penalizes these unrealistic choke-points. 

Crucially, this enforces **Continuous Pathing**: ProcTHOR's NavMesh baker physically simulates continuous traversal. It proves that there is an unbroken, flowing path between points, rather than just disjointed open squares on a grid that a discrete array check might falsely validate.

*(Note: In our baseline vs. guidance models, we observed that forcing realistic spacing naturally consumes more free space, resulting in realistic walkability scores around ~20% for dense bedrooms).*

---

## 3. The Navigability (Accessibility) Metric

The second, and arguably more critical metric, is **Navigability (Accessibility)**. While Walkability evaluates free space, Navigability evaluates the **functional placement of objects relative to that free space**.

### How it Works
The script takes the exact list of valid NavMesh standing points generated by ProcTHOR and cross-references them against every single piece of furniture generated in the scene (desks, beds, chairs, lamps).

### The Mathematical Threshold
- For each generated object, the physics engine calculates the physical distance from the object's collision edge to the closest valid NavMesh standing point.
- If there is a standing point within **$0.75m$** of the object's edge, the object is marked as "Accessible."
- **Accessibility Rate = (Accessible Objects) / (Total Objects)**

### Why it Matters
This is the ultimate ground-truth verification of our layout guidance. In a purely python-based heuristic model, a desk placed directly behind a massive wardrobe might trigger zero "collision" loss, but it is fundamentally unusable by a human. Our Navigability metric catches this. 

If an object is trapped, barricaded, or spawned in a corner with no physical path leading to it, the agent cannot get within $0.75m$, and it fails the accessibility check.

## 4. Fair Evaluation via Post-Process Collision Resolution

A major flaw in naive floor-space evaluation is that models generating heavily overlapping objects (or objects clipping into walls) will artificially report **higher Walkability**. Because two overlapping beds take up significantly less physical footprint than two properly separated beds, a failing model is unfairly rewarded with extra "free space."

To solve this and ensure a fair metric comparison, we introduce a **post-processing collision resolution step** (`resolve_bbox_collisions_obb`) using the Separating Axis Theorem (SAT). 
- Before calculating ProcTHOR metrics, this algorithmic pass pushes overlapping Oriented Bounding Boxes (OBB) apart until they no longer intersect. 
- It forces all generated objects to occupy their true, uncompromised physical space in the room.

It is only reasonable to evaluate Walkability and Navigability on a scene where objects do not collide anymore. By enforcing this post-process separation, poorly optimized baselines are forced to spread their objects out, which realistically devours their artificially inflated "Walkability" space and accurately exposes choke-points. This guarantees that our evaluation metrics reflect true functional layout capabilities rather than rewarding impossible physics.

## Summary of Impact

By implementing this ProcTHOR verification suite and our collision-resolution pre-step, we successfully prove the real-world viability of our generative layout model. 

While the Python-based `physical_guidance` operates as the generative engine pushing the model toward better layouts, the **ProcTHOR Walkability and Navigability metrics provide the empirical, physics-backed evidence** that our contribution fundamentally improves the quality, logic, and human-usability of the generated 3D environments. Our evaluations show a marked improvement in functional accessibility, proving that our generation isn't just mathematically optimized—it is physically sound.

## Appendix: Engineering & Implementation Details

While the academic contribution focuses on continuous 2D pathfinding and functional accessibility, the actual translation from a generative model output to a 3D Unity engine requires specific engineering workarounds:

- **NavMesh "Leaking" Fix (Ceiling Extrusions):** By default, if furniture is represented simply as thin 2D walls or low-profile bounds, Unity's volumetric NavMesh baker (which casts voxels downwards) can leak through the corners or over the tops, artificially allowing the agent to teleport or cut through solid objects. To enforce a perfect 2D pathing constraint, our conversion script dynamically extrudes every furniture footprint into a ceiling-high ($3.0m$) solid pillar, capped with a horizontal lid. This guarantees an impenetrable volume that the engine perfectly respects.
- **Agent Spawn Safety (EDT):** Before the physics engine can evaluate NavMesh points, the agent must spawn in the room without clipping into geometry (which would crash the simulation). We implement a 2D Euclidean Distance Transform (EDT) via `scipy` on the rasterized floor plan to mathematically guarantee the agent teleports to the safest, widest open coordinate in the room before beginning the evaluation.
