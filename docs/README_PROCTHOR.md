# ProcTHOR Evaluation Scripts

This directory contains documentation and instructions for evaluating EchoScene layouts using AI2-THOR / ProcTHOR.

## Dependencies
Before running these scripts, ensure you have the required packages installed:
```bash
pip install ai2thor prior numpy matplotlib scipy
```

## Files and Scripts
All scripts are located in the `scripts/procthor_eval` directory and should be run from the main `echoscene` directory.

1. **`scripts/procthor_eval/convert_echoscene_to_procthor.py`**: Reads `physcene_collision_input.json` from the generated outputs, maps the bounding boxes to ProcTHOR room boundaries, and converts the furniture into solid walls (with ceiling lids) to guarantee exact collision sizes in Unity.
2. **`scripts/procthor_eval/eval_walkability.py`**: Computes the ratio of empty floor space (NavMesh) relative to total room area. Note that overlapping objects artificially inflate this score.
3. **`scripts/procthor_eval/eval_navigation.py`**: Computes **Accessibility** by spawning an AI2-THOR agent and testing if every single generated object is reachable within 0.75m. This is a 2D equivalent of ProcTHOR's native `GetInteractablePoses` metric.
4. **`scripts/procthor_eval/plot_path.py`**: Visualizes the AI2-THOR native `GetShortestPathToPoint` routing around the generated layouts.
5. **`scripts/procthor_eval/run_convert.sh`**: Helper shell script to convert `baseline`, `physcene_guidance`, and `released_full_model` simultaneously.
6. **`scripts/procthor_eval/run_all_evals.py`**: Python orchestrator to run walkability, accessibility, and plotting across all datasets.
## Metric Definitions: Walkability vs. Navigability
Both metrics rely on the exact same core feature of ProcTHOR: the **NavMesh** (`GetReachablePositions`). However, they use that data to answer two completely different questions about the quality of the generated room:

### 1. Walkability (How much of the room is usable?)
* **How it works**: ProcTHOR floods the floor and returns a list of every single valid `0.25m` grid point where the agent can physically stand without colliding with anything (accounting for the `~0.3m` agent radius). 
* **The Calculation**: The script counts up all those points, multiplies them by the area of a single cell (`0.25 * 0.25 = 0.0625 m²`), and divides it by the total mathematical footprint of the room.
* **The Goal**: It measures if the diffusion model generated a spacious, easily traversable room, rather than a cramped layout entirely filled by overlapping furniture.

### 2. Navigability / Accessibility (Can the agent reach the furniture?)
* **How it works**: The script takes that exact same list of valid standing points generated by ProcTHOR. But instead of just counting them, it iterates through **every single piece of furniture** generated in the scene (desks, chairs, lamps, etc).
* **The Calculation**: For each object, it calculates the physical distance from the object's edge to the closest valid NavMesh point. If there is a standing point within **`0.75m`** of the object, the object is marked as "Accessible". 
* **The Goal**: It measures the percentage of objects that an embodied agent could actually walk up to and use. For example, if a chair is completely trapped behind a bed and a wardrobe, the agent can't get within `0.75m` of it, so it fails the accessibility check!

In short, **Walkability** evaluates the free space, while **Navigability** evaluates the functional placement of objects relative to that free space.

### 3. Agent Teleportation (How do we find a safe starting point?)
Before any AI2-THOR NavMesh metric can run, the agent must be successfully teleported into the room. If the agent spawns inside a wall or a piece of furniture, the physics engine will throw a fatal `InvalidOperationException`.
* **The Solution (EDT)**: The evaluation scripts use a 2D Euclidean Distance Transform (EDT) via `scipy.ndimage.distance_transform_edt` on the rasterized floor plan.
* **How it works**: It calculates the physical distance from every free floor pixel to the absolute nearest obstacle (furniture or wall). The pixel with the **maximum distance** (the safest, most open spot in the room) is chosen as the initial spawn coordinate.
* **The Result**: This mathematical guarantee ensures the AI2-THOR agent is teleported perfectly into the widest available clearing, completely eliminating out-of-bounds spawn crashes!

## How to run the pipeline
To evaluate a new generation run (e.g. `output/physcene_guidance`):

1. **Convert to ProcTHOR JSON**:
   ```bash
   python scripts/procthor_eval/convert_echoscene_to_procthor.py --full --bbox_path ./output/physcene_guidance/vis/2050/physcene_collision_input.json --out_dir ./output/physcene_guidance/vis/2050/procthor_scenes
   ```
2. **Run Metrics**:
   ```bash
   python scripts/procthor_eval/eval_walkability.py --scenes_dir ./output/physcene_guidance/vis/2050/procthor_scenes
   python scripts/procthor_eval/eval_navigation.py --scenes_dir ./output/physcene_guidance/vis/2050/procthor_scenes
   ```

3. **Visualize Navigation Paths**:
   To generate a visual plot of the layout overlaid with the 0.05m raster grid, AI2-THOR's native NavMesh reachable points, and the shortest path between the EDT-selected spawn point and the furthest valid position:
   ```bash
   python scripts/procthor_eval/plot_path.py --scene_name SecondBedroom-6482 --folders baseline physcene_guidance released_full_model
   ```

The metric results are saved directly in the `procthor_scenes` folder as `walkability_results.json` and `navigation_results.json`. The generated plots are saved alongside the `physcene_collision_input.json` in the respective generation output folders.
